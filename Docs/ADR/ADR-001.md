# ADR-001: Visión y decisiones fundamentales del proyecto Quintana

**Fecha:** 2025-06-30  
**Estado:** Aprobado  
**Autor:** Claudia / Equipo Quintana CABA Propiedades  
**Contexto:** Antes de implementar nuevas funcionalidades, se documentan las decisiones fundamentales del proyecto para que cualquier desarrollador o IA entienda la visión antes de escribir código.

---

## 1. Visión del proyecto

**Quintana CABA Propiedades** es una inmobiliaria que comienza su operación en Capital Federal. El sitio web debe cumplir un objetivo comercial primario: **captar propiedades para comercializar**. En el negocio inmobiliario, primero se consiguen propiedades para vender o alquilar, y luego se encuentran compradores o inquilinos. Por lo tanto, la web debe orientarse a convencer a los propietarios de confiar en Quintana para la tasación, publicación y comercialización de su propiedad.

**Objetivo comercial principal:** Convertir visitantes en propietarios que soliciten una tasación sin cargo.

**Objetivo secundario:** Mostrar propiedades disponibles para compradores e inquilinos interesados.

**Mensaje principal:** "Vendé o alquilá tu propiedad en CABA con acompañamiento profesional y tecnología."

---

## 2. Identidad institucional y roles

- **Marca institucional:** Quintana CABA Propiedades. El sitio refuerza la marca como equipo, no como una sola persona.
- **Carina:** Referente institucional y dueña de la inmobiliaria.
- **Claudia:** Asesora inmobiliaria clave dentro del equipo, con visibilidad progresiva y profesional.
- **Regla de comunicación:** la web no debe presentar a Claudia como la única cara de la empresa. Los CTAs institucionales priorizan "Contactar al equipo", "Solicitar asesoramiento" o "Quintana".
- **Fotos personales:** no se publican fotos de Carina o Claudia sin autorización explícita. Se usan placeholders hasta contar con el OK.

## 3. Decisiones arquitectónicas clave

### 3.1. Tecnología del frontend
- **Framework:** React + Vite + TypeScript.
- **Estilos:** Tailwind CSS.
- **Enrutamiento:** React Router DOM.
- **Build:** Vite genera el sitio estático en `dist/`.
- **Razón:** stack moderno, rápido de mantener y con buen rendimiento para SEO.

### 3.2. Tecnología del backend
- **Framework:** Node.js + Express.
- **Servidor de archivos estáticos:** Express sirve `dist/` y `public/`.
- **API:** endpoints REST para admin, fotos, leads, Instagram, etc.
- **Razón:** simple, suficiente para el MVP y fácil de deployar en Hostinger.

### 3.3. Persistencia de datos
- **Propiedades:** archivos de texto (`datos_raw.txt`, `descripcion.txt`) y `properties.json` generado por scripts Python.
- **Configuración:** `fichas/config.json` (no subir a GitHub si contiene secrets).
- **Fotos:** carpetas `fichas/{id}/Imagenes/` y copia a `public/properties/{id}/images/`.
- **Leads futuros:** SQLite para CRM liviano (ver tickets LEAD-001 y LEAD-003).
- **Razón:** la carga actual es baja, el equipo conoce el flujo de archivos y el scraping se integra al ecosistema de fichas.

### 3.4. Scraper y automatización
- **Origen de datos:** Zonaprop via ScrapingBee + OpenAI.
- **Scripts:** Python (`agente.py`, `update_web.py`, `rewriter_descripciones.py`, etc.).
- **Audio:** ElevenLabs para narraciones de propiedades.
- **Instagram:** generación manual de tarjetas y captions; publicación automática pendiente de credenciales de Meta.
- **Razón:** automatizar la carga inicial de propiedades y contenido, manteniendo revisión humana antes de publicar.

### 3.5. Deploy
- **Producción:** Hostinger VPS con Node.js y Passenger.
- **Dominio:** https://quintana.cabapropiedades.ar
- **Flujo:** build local → ZIP via `scripts/zip_site.py` → subida SFTP → restart de Passenger.
- **Razón:** infraestructura ya contratada y operativa; el flujo ZIP es seguro y controlado.

## 4. Decisiones de UX / producto

- **Home:** landing orientada a captar propietarios, con Hero claro, beneficios, formulario de tasación y propiedades destacadas más abajo.
- **CTA principal:** "Solicitar tasación".
- **CTA secundario:** "Ver propiedades".
- **WhatsApp:** siempre visible como botón flotante, con mensaje prearmado según contexto (propiedad, tasación, contacto general).
- **Mobile first:** todos los diseños se validan primero en mobile.
- **SEO:** title, description, Open Graph, Schema.org, sitemap y robots.txt configurados por página.
- **Performance objetivo:** Lighthouse Performance > 90, SEO > 95.

---

## 5. Decisiones de seguridad y privacidad

- **Secrets:** `.env`, `fichas/.env`, cookies, tokens de Instagram y configuraciones sensibles **nunca se suben a GitHub**.
- **Archivos grandes:** `node_modules/`, `dist/`, `site_express.zip`, mapas y fotos originales no se versionan.
- **Copias de seguridad:** antes de refactors importantes se crean backups en `backups/<timestamp>/`.
- **Leads:** futura base SQLite con datos de contacto; no se exponen públicamente ni se versionan.

---

## 6. Flujo de trabajo de desarrollo

1. Todo cambio comienza con un ticket en `tickets.json`.
2. Antes de grandes cambios se crea backup en `backups/`.
3. Se desarrolla y prueba en `http://localhost:5173`.
4. Se ejecuta `npm run build` para verificar TypeScript y generar sitemap.
5. Se sube a GitHub con commit claro.
6. Solo después de validación local se deploya a producción en Hostinger.
7. Se actualiza `avances.md` con resumen y notas de verificación.

---

## 7. Criterios de éxito del proyecto

- La home comunica en menos de 5 segundos qué hace Quintana y cómo solicitar una tasación.
- El formulario de tasación genera consultas reales por WhatsApp.
- El panel admin permite gestionar propiedades, fotos, portada y scraping sin depender de desarrolladores.
- El sitio carga rápido en mobile y desktop.
- El SEO local posiciona búsquedas de tasación, venta y alquiler en CABA.
- El sistema es mantenible por otra IA o desarrollador con esta documentación.

---

## 8. Notas para futuros desarrolladores / IA

- La carpeta `fichas/` es la fuente de verdad de propiedades. `properties.json` y `public/properties/` se regeneran desde `fichas/`.
- `fichas/web-propiedades/server.js` detecta automáticamente la ruta de `fichas/` según dev o producción.
- Antes de tocar `server.js`, `App.tsx` o `Home.tsx`, leer este ADR y los tickets relacionados.
- Si una nueva funcionalidad impacta la marca o la comunicación institucional, consultar `tickets.json` (GOV-001, BRAND-001, CONV-001) antes de implementar.
- Nunca publicar fotos, nombres o datos personales sin autorización explícita del equipo Quintana.
